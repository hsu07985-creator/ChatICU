## 2. 階段註解

### 2.1 A1 raw — 破了 OpenAI cache

第一版 critical/deferred split (commit `c97a5e5c5`) 用 `_merged_snapshot()` 把 deferred 文字 append 進 system_prompt。看似乾淨，**但 system_prompt 跨 turn 變動 = OpenAI prompt cache prefix 不再 byte-stable**。

實測：cache_hit_ratio_p50 從 baseline 70% 跌到 0%。subsequent-turn TTFT 損失 cache、整體變慢。

→ **A1 的 first-turn 省 2 秒，但 subsequent turn 全失 cache**，淨效果不明朗甚至虧。

### 2.2 A1.1 — 把 deferred 移出 system_prompt

修法（commit `458a70260`）：
- 刪除 `_merged_snapshot()`
- system_prompt 永遠 = TASK_PROMPTS + critical-only snapshot（byte-stable）
- deferred 改成注入 ephemeral `user_message`（LLM 看得到、但不寫入 DB history、不進 cache prefix）

新增 9 個 unit test 守 invariant：
- `system_prompt_byte_stable_across_turns`
- `system_prompt_does_not_include_deferred_text`
- `system_prompt_unchanged_regardless_of_deferred_state`
- 5 個 edge-case skip-injection 測試
- 1 個 happy-path 測試

實測：cache_hit_ratio_p50 從 0% 回到 55%。`ttft_ms` p95 +10.8%（在 acceptance tolerance 內）。

→ A1.1 是**淨正向**：first-turn 省 2s、cache 大致回穩。

### 2.3 B — Multi-connection true parallel

A1.1 後 snapshot_ms p95 = 4103ms，仍然在 partial-success（3000-4500）區間，沒打到 < 3000ms target。

`b15-snapshot-audit-2026-04-30.md` 確認：6 個 fetcher 共用 1 個 AsyncSession，在 asyncio.gather 下只有 ~30-40% parallel efficiency（actual 5s vs sum 17s）。

修法（commit `4919d4644`）：給 `build_critical_snapshot` 每個 fetcher 開獨立 AsyncSession (`_async_session()`)，asyncpg connection 級別真平行。

實測：snapshot_ms p95 從 4103ms 降到 **3835ms** —— 只省 ~268ms（−7%）。**B 的收益遠低於預期**。

可能原因：
- Supabase pooler 連線取得 overhead（每 conn ~200ms × 6 = ~1.2s）
- SSL / connection warmup 在 fresh session 上有成本
- `pool_size=5 + max_overflow=5 = 10` 可能 throttle 6 並發

`ttft_ms` p50 從 1321 → 920（−30%），但這變化不可歸因於 B（B 不影響 LLM 端），更像 OpenAI 端 noise。

→ **B 給了少量 wins，沒打到 target**。`<3000ms` 看起來不是 multi-connection 能單獨解決的。

## 8. 重啟時的具體下一步（**D5: single-connection bulk SELECT**）

如果上面條件成立，**從 D5 動工**而不是再做 cache：

### Why D5
- A1.1+B canary 證明 multi-connection 只省 7%（4103 → 3835）
- 瓶頸不在 connection 數量、而在 6 個獨立 SELECT 的 RTT 累積（即使並行）
- 把 6 個 SELECT 合成 1 個 multi-table UNION 或 single CTE → 省 5 個 RTT

### How D5 大致
```sql
WITH p AS (SELECT ... FROM patients WHERE id = $1),
     l AS (SELECT ... FROM lab_data WHERE patient_id = $1 ORDER BY ts DESC LIMIT 1),
     l24 AS (SELECT ... FROM lab_data WHERE patient_id = $1 AND ts <= ... LIMIT 1),
     m AS (SELECT ... FROM medications WHERE patient_id = $1 AND status='active'),
     v AS (SELECT ... FROM vital_signs WHERE patient_id = $1 ORDER BY ts DESC LIMIT 1)
SELECT (SELECT row_to_json(p) FROM p) as patient,
       (SELECT row_to_json(l) FROM l) as lab,
       ...
```

預期 build_ms：1 RTT × ~0.5s（Supabase Sydney）+ DB 平行 query work ≈ **600-1000ms**，打進 < 2000ms 目標。

### D5 的風險
- 寫 raw SQL 失去 ORM 型別安全
- duplicate detection 仍需要 medication ORM objects（need separate fetch path）
- single CTE 會 lock 多 table、慢 query 影響面大
